Levels of data modeling

An enterprise is closely connected with people, place, animations, or things, and their interests. A data model provides a visual representation of these aspects. In addition to providing visual cues, a data model acts as a communication guideline between technical and non-technical stakeholders, such as people. Since a data model is composed of symbols that illustrate the concepts that should be communicated and agreed upon, it is often termed the blueprint of the data. These so-called blueprints are constructed at several distinct levels and details, ranging from high-level requirements, to basic architectural layout requirements, to detailed design specifications. These blueprints (or models) can be constructed at different levels of design, referred to as levels of data modeling. There are three levels of data modeling: conceptual data modeling, logical data modeling, and physical data modeling. 

The conceptual data model (CDM) resides at the top, providing a satellite view of the business solution. Also, the CDM gives a context and scope to the logical data model (LDM), which produces a detailed business solution. Conceptual data modeling

Conceptual data modeling is the first step in the top-down approach, where the goal is to capture a satellite view of the business requirements of an organization. As has been claimed by many authors, the main reason for conceptual data modeling is to capture the big picture and comprehend the scope of the high-level requirements of an organization's project. It is worth noting that CDMs are created as a precursor to an LDM or as an alternative to it. 

Conceptual data modeling tries to answer the following questions:

  • What problems are involved with the business and require immediate solutions? 
  • What are the core concepts of these problems?
  • How are these problems related to one another?
  • Is there any scope of the effort? What is the scope in relation to the effort made?

The conceptual data modeling process results in a CDM. The model that is created illustrates the key concepts required for a particular application's development. In addition, the model provides a brief overview of the effort required by the company to construct the software:

The conceptual model comprises the most important concepts of the project and requirements that each of those concepts can be defined, and documented. To summarize, the CDM explores the topmost relationships between several entities. Hence, this type of model can be referred to as a domain model. 

Logical data modeling

An LDM is utilized as the initial step of the physical data model. The LDM consists of the detailed structures of a system. In addition, the LDM depicts the detailed relationships between the structures. An LDM can be described as detailed if it contains all of the essential entities, attributes, and key groups, and the relationships they have with one another. The LDM does not rely on the types of DBMSs used. So, the models should look the same regardless of the type of DBMS system, such as MongoDB, Oracle, and so on.

Benefits of constructing LDMs

The following list describes the main benefits of constructing LDMs:

  • It facilitates an understanding of business data elements and their requirements
  • It provides a foundation for designing the database
  • It helps to avoid data redundancy and prevents data and business inconsistency
  • It promotes data reusability and sharing
  • It diminishes development, maintenance cost, and time consumption
  • It confirms a logical process model and helps to impact analysis

The LDM provides great mapping, in the case of multiple technologies. The LDM will explain one technology-independent perspective, and then we can map this business perspective to each of the various technology perspectives. Logical data modeling involves completing four steps, including filling in the CDM, normalizing and abstracting the model, determining the most useful form, and finally reviewing and confirming the logical model. The most common features of a logical model involve the following:

  • All entities and relationships among the model are given
  • All attributes are specified for each entity
  • The primary key is specified for each entity
  • Foreign keys are specified
  • Normalization occurs at this level

Physical data modeling

A physical data model illustrates how the different elements will be constructed, as per business requirements that were provided in the LDM. It gives a visual illustration of the physical structure of an actual database. It may also include components of the LDM that have been refactored, in order to optimize the database and enhance the application's performance. In the physical data model, objects are defined at a schema level. Here, a schema can be defined as a group of related objects in a database. To summarize, a model can be referred to as a physical data model if it contains all table structures, including column names, column data types, primary keys, column constraints, and the relationship between tables. Features of the physical data model

As mentioned previously, a physical data model shows the way in which the database is built, using the model. In this section, you will learn about the various features of an LDM. Some noticeable features of a physical data model are given as follows:

  • The model represents all tables and columns
  • The physical model contains all of the foreign keys that are utilized to recognize relationships between tables
  • The physical data model varies for different RDBMS
  • Based on the user requirements, denormalization can occur in this model

When logical data modeling is complete, it is expected to produce a server model diagram that shows tables, columns, keys, and relationships within the database. In addition, user feedback documentation and database design documentation are expected to be produced.   • The XMS schema
  • Legacy data
