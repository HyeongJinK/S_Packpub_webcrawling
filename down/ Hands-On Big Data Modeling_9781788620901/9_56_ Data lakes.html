Data lakes

With big data streaming from various origins, in varying formats and models, and at various speeds, it is no surprise that we need to be able to ingest this data into a fast and scalable storage system that is flexible enough to serve many (current and future) analytical processes. When it comes to such a dynamic storage system, traditional storage systems don't fit big data challenges for streaming and batch processing applications. That is where the concept of data lakes comes in.

We can store all structured and unstructured data at various scales, in a centralized repository. Such a repository is referred to as a data lake. We can store our data as it is, without having to structure or preprocess the data. In addition to that, we can execute various types of analytics and visualizations, and even real-time analytics.

Differences between data lakes and data warehouses

The primary purpose of both data warehouses and data lakes is storage. However, the difference between both lies in the way that data is stored. This relates to the way that they allow access to end users, and the way that a schema is created, updated, and destroyed.

Note

To analyze some common differences between data warehouses and data lakes, refer to https://aws.amazon.com/big-data/what-is-a-data-lake/.

It takes a lot of time to analyze a data source and understand the business process profile when developing a data warehouse. This process results in a structured data model. In a data lake, all of the data is retained, no matter of how much is used. This data is made available to users, who users can consume the data at any point in time, and can perform business analysis on it. Data lakes support every type of data. 

How a data lake works

The following diagram shows how a typical data lake works. The data gets loaded from its source. Then it stores the data in its native format, until it is required. Once the data is required, the application can freely read the data and add a structure to it. This is what we refer to as schema-on-read:

In a traditional data warehouse, the data is loaded into the warehouse after being transformed into a well-defined format. This is what we call schema-on-write. Any application that uses the data needs to know this format. In this approach, data is not loaded into the warehouse unless there is a use for it. However, the schema-on-read approach of data lakes ensures that all of the data is stored for use at a later time.
