Types of data model

You should now understand that data models are very important in the construction of a real database. Most DBMS are constructed in a structural fashion. These models can be categorized into different types, based on their structures. Different models apply to different stages of the database design process. In this section, we will explore some of the most common types of data model that are used in real business scenarios.

Hierarchical database models

A hierarchical model uses a tree-like structure to represent the data, with a single parent, or a root, in each record. The record corresponds to a single table, holding information about a particular subject, and is connected to other records using links. A specific order is used to sort sibling records.

Relational models

Relational models sort data into tables that are known as relations. Columns and rows make up the relations. Each row holds a unique value, called a tuple. Each column holds a value, called an attribute, from the same area:

  • A domain is made up of the attributes of a relation
  • A primary key is a specific attribute, or a combination of attributes
  • A primary key in another table is called a foreign key
  • A tuple includes the specific data of an entity, such as a particular employee
  • The degree of the relation denotes the number of attributes or fields in the relation
  • The cardinality is the number of tuples in a relationship

A relational model is a declarative method that can specify data and queries. This means that users directly state what information is in the database and the information that they need, and let the database management system software define the data structures for saving the data and the retrieval methods for answering queries:

Advantages of the relational data model

The relational data model is very popular and is a common type of data model employed in the database management ecosystem. Some significant benefits of using a relational data model are as follows:

  • The main benefit of this model is its capability to describe data in an uncomplicated format
  • The process of managing records is simplified through the use of certain key attributes applied to retrieve the data
  • The representation of different types of relationship is possible with this model

Network models

The network data model addresses some of the issues faced with hierarchical data models. In a network data model, the data is represented in terms of nodes and a set of links. Two basic terms to understand are as follows:

  • A node represents a collection of records. A record is analogous to an entity in the ER model. Each node representing a record is, in fact, a collection of fields or attributes that holds only one data value. 
  • The relationship between one record and another record is shown by links.

Entity-relationship models

Entities in the entity-relational consist of stored data points on people, places, and things, each having certain attributes that constitute their domain. The cardinality, or the relationships between entities, is also mapped. The entity relationship model is based on the notion of real-world entities and their relationships. An entity set is created while formulating the real-world scenario into the database model that is dependent on two vital features, as follows:

  • Entities and their attributes: For example, a student entity may have a name, class, and age as attributes.

Object-relational models

The object-relational model is a hybrid database model that is an amalgamation of the primitiveness of the relational model with some of the superior functionality of the object-oriented database model. In essence, it enables designers to combine objects into a familiar table structure. There are several languages and call interfaces for the object-relational model. Some of the most common examples include SQL3, vendor languages, ODBC, and JDBC. 

The following list shows some advantages provided by the object-relational model:

  • It supports many geometry types (including arcs, circles, compound polygons, compound line strings, and optimized rectangles), which gives it more flexibility
  • It provides an ease of use in building and managing indexes, and in performing spatial queries
  • It has index maintenance from the Oracle database
  • Its geometry is modeled in a single column
  • It provides optimal performance

Let's compare how a relational query differs from the object-relational database model. In order to create a table that stored patient information in RDBMS, our SQL query would be something like the following:

CREATE TABLE patients (
    id    CHAR(12) NOT NULL PRIMARY KEY,
    name    VARCHAR(100) NOT NULL,
    email    VARCHAR(40) NOT NULL,
    DOB    DATE    NOT NULL,
    phone    VARCHAR(13)
)

To select any patient's information from the table, our SQL query would look as follows:

SELECT name FROM patients WHERE Month(DOB) = Month(getUserDate()) AND Day(DOB) = Day(getUserDay())

In an object-relational database, something similar can be achieved via the following SQL statements:

CREATE TABLE patients (
 id Patient_ID NOT NULL PRIMARY KEY,
 name PatientName NOT NULL,
 email PatientEmail NOT NULL,
 DOB DATE NOT NULL,
 phone VARCHAR(13)
)

The corresponding query would be as follows:

SELECT Formal (P.id) FROM patients P where Birthday(P.DOB) = TODAY;

The object-relational model is complex, yet it has very powerful features. Based on the efficiency of the results of this model, its complexity can be subsided. The object-relational model can easily work with other models, including the relational model or the network model.
